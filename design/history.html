01. 이력(히스토리) 테이블 설계 — 스냅샷 vs AS-IS/TO-BE

"변경 이력을 남겨주세요"라는 한 문장 뒤에는 서로 다른 세 가지 요구 — 감사(audit), 사용자 활동 내역, 시점 조회 — 가 숨어 있습니다. 스냅샷 방식과 AS-IS/TO-BE(변경분) 방식의 구조적 차이를 이해하고, 용도별로 올바른 설계를 고르는 기준과 흔한 안티패턴을 정리합니다.

중급 ⏱ 약 30분 🎬 애니메이션 2개 ✍️ 퀴즈 3개 예제: PostgreSQL 문법
🧭
이 페이지의 구성

① 스냅샷 vs AS-IS/TO-BE 구조 비교 ② 실무 패턴 3가지(jsonb 하이브리드 · 유효기간/SCD2 · 도메인 이벤트) ③ "누가 보는 이력인가"에 따른 용도별 선택 기준 ④ 안티패턴 6선 ⑤ 설계 체크리스트. 예제는 PostgreSQL 문법이지만 개념은 모든 RDBMS에 통합니다.

🥇 제1원칙

용도가 설계를 결정한다. 감사·사용자 노출·시점 조회는 요구가 다르다. 하나의 이력 테이블로 전부 해결하려는 순간 어느 용도에도 최적이 아니게 된다.

🥈 제2원칙

이력은 append-only. UPDATE/DELETE가 필요해지는 이력 설계는 어딘가 잘못된 것이다. 수정 가능한 이력은 증거가 아니다.

🥉 제3원칙

보존 정책은 처음부터. 이력 테이블은 반드시 원본보다 빨리 커진다. 파티셔닝과 아카이브 계획 없이 시작하면 1~2년 뒤 운영 장애로 돌아온다.

1. 두 방식 — UPDATE 한 번이 어떻게 기록되는가

주문의 상태가 PAID → SHIPPED로 바뀌는 UPDATE 한 번을 두 방식이 각각 어떻게 기록하는지부터 보겠습니다. 구조 차이가 곧 강점/약점의 차이입니다.

주문 1042번이 결제 완료(PAID) 상태로 존재합니다. 이제 이 행에 UPDATE가 한 번 발생합니다.

UPDATE orders SET status='SHIPPED' — 바뀐 것은 status 컬럼 하나뿐입니다. 이 변경을 두 방식이 각각 어떻게 남기는지 봅시다.

🅰 스냅샷 방식은 변경 후(또는 전) 행 전체를 통째로 복사해 이력 테이블에 추가합니다. 바뀌지 않은 amount, addr까지 전부 다시 저장됩니다.

🅱 AS-IS/TO-BE 방식은 바뀐 컬럼의 이전 값(AS-IS)과 이후 값(TO-BE)만 기록합니다. 저장량은 훨씬 적지만, old/new를 비교해 diff를 뽑는 로직이 필요합니다.

강점이 정확히 반대입니다. "그 시점의 전체 상태"는 스냅샷이 행 하나로 답하고, "무엇이 바뀌었나"는 AS-IS/TO-BE가 기록 그 자체로 답합니다. 스냅샷에서 diff를 얻으려면 이전 행과 비교 연산이, delta에서 시점 복원을 하려면 변경분을 순서대로 재적용하는 작업이 필요합니다.

1-1. 🅰 스냅샷 방식 (Full Row Snapshot)

-- 원본과 거의 동일한 구조 + 이력 메타데이터
CREATE TABLE orders_history (
    history_id   bigserial PRIMARY KEY,
    order_id     bigint NOT NULL,     -- 원본 PK (FK는 걸지 않는다 — 안티패턴 ② 참고)
    status       text,
    amount       numeric,
    addr         text,
    -- … 원본 컬럼 전부 …
    operation    char(1) NOT NULL,    -- I / U / D
    changed_by   bigint,
    changed_at   timestamptz NOT NULL DEFAULT now()
);

-- "그 시점의 상태" 복원: 해당 시각 직전의 마지막 스냅샷 한 건
SELECT * FROM orders_history
WHERE  order_id = 1042 AND changed_at <= '2026-07-14 14:00'
ORDER BY changed_at DESC LIMIT 1;

1-2. 🅱 AS-IS/TO-BE 방식 (변경분 기록, Delta)

-- 컬럼 단위로 쪼개 기록하는 형태가 가장 흔하다
CREATE TABLE orders_change_log (
    log_id       bigserial PRIMARY KEY,
    order_id     bigint NOT NULL,
    column_name  text NOT NULL,       -- 바뀐 컬럼
    before_value text,                 -- AS-IS (타입이 text로 뭉개지는 점에 주의)
    after_value  text,                 -- TO-BE
    changed_by   bigint,
    changed_at   timestamptz NOT NULL DEFAULT now()
);

-- "오늘 누가 뭘 바꿨나": 기록 자체가 답
SELECT changed_at, changed_by, column_name, before_value, after_value
FROM   orders_change_log
WHERE  order_id = 1042
ORDER BY changed_at;

1-3. 트레이드오프 한눈에

관점🅰 스냅샷🅱 AS-IS/TO-BE
특정 시점 상태 복원매우 쉬움 — 그 시점 행 하나 조회어려움 — 변경분을 순서대로 재적용
"뭐가 바뀌었나" 조회어려움 — 이전 행과 컬럼별 비교 필요매우 쉬움 — 기록 자체가 diff
저장 공간낭비 큼 — 한 컬럼 바뀌어도 전체 복사효율적 — 바뀐 것만 저장
스키마 변경 대응원본 컬럼 추가 시 이력 테이블도 동시 변경유연 — 컬럼명이 데이터라서 영향 없음
타입 안정성원본과 동일한 타입 유지before/after가 text/jsonb — 타입 정보 유실
기록 로직단순 — 행 복사 INSERT복잡 — old/new 비교해 diff 추출
감사(audit) 신뢰성높음 — 그 시점의 완전한 증거diff 추출 로직에 버그 있으면 기록 누락 위험
💡
한 줄 요약

스냅샷은 "그때 상태가 뭐였나"에 강하고, AS-IS/TO-BE는 "누가 뭘 바꿨나"에 강합니다. 어느 질문을 더 자주 받게 될지가 선택의 출발점입니다.

2. 실무 패턴 3가지

2-1. 하이브리드 — jsonb 스냅샷 + 변경 컬럼 목록 (감사 로그의 사실상 표준)

PostgreSQL에서 감사 로그 용도로 가장 널리 쓰이는 패턴입니다. 행 전체를 jsonb로 스냅샷 뜨면서(old/new 둘 다), 바뀐 컬럼 목록도 함께 저장합니다. 두 방식의 장점을 모두 가져가고, 트리거 하나로 모든 테이블에 범용 적용됩니다.

CREATE TABLE audit_log (
    id           bigserial PRIMARY KEY,
    table_name   text NOT NULL,
    record_id    bigint NOT NULL,
    operation    text NOT NULL,          -- INSERT / UPDATE / DELETE
    old_data     jsonb,                  -- AS-IS 행 전체
    new_data     jsonb,                  -- TO-BE 행 전체
    changed_cols text[],                 -- 바뀐 컬럼만 표시
    changed_by   bigint,
    changed_at   timestamptz NOT NULL DEFAULT now()
) PARTITION BY RANGE (changed_at);     -- 보존 정책은 처음부터! (제3원칙)

-- 범용 감사 트리거 — 어떤 테이블에든 그대로 부착 가능
CREATE OR REPLACE FUNCTION fn_audit() RETURNS trigger AS $$
BEGIN
  INSERT INTO audit_log
        (table_name, record_id, operation, old_data, new_data, changed_cols, changed_by)
  SELECT TG_TABLE_NAME,
         COALESCE(NEW.id, OLD.id),
         TG_OP,
         to_jsonb(OLD),
         to_jsonb(NEW),
         CASE WHEN TG_OP = 'UPDATE' THEN
           (SELECT array_agg(key) FROM jsonb_each(to_jsonb(NEW)) n
             WHERE n.value IS DISTINCT FROM to_jsonb(OLD) -> n.key)
         END,
         nullif(current_setting('app.user_id', true), '')::bigint;
  RETURN NULL;   -- AFTER 트리거
END $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_orders_audit
AFTER INSERT OR UPDATE OR DELETE ON orders
FOR EACH ROW EXECUTE FUNCTION fn_audit();
💡
"누가"를 트리거에 전달하는 법 — set_config 패턴

트리거는 DB 세션만 알 뿐 앱의 로그인 사용자를 모릅니다. 트랜잭션 시작 직후 앱에서 SELECT set_config('app.user_id', '42', true)를 실행하면(세 번째 인자 true = 트랜잭션 한정) 트리거가 current_setting('app.user_id', true)로 읽을 수 있습니다. 위 예제가 이 패턴입니다.

2-2. 유효기간 방식 — SCD Type 2, "시점 조회"의 왕

요금·환율·계약 조건처럼 "3월 15일 기준 가격"을 정확히 재현해야 하는 데이터는 별도 이력 테이블보다 본 테이블에 유효기간을 두는 편이 낫습니다. 변경 시 기존 행의 기간을 닫고 새 행을 추가합니다 — 데이터 웨어하우스에서 말하는 SCD(Slowly Changing Dimension) Type 2와 같은 구조입니다.

상품 77의 가격 행입니다. 유효기간의 끝이 열려 있으면(∞) "현재 유효한 가격"이라는 뜻입니다.

4월 1일부터 18,000원으로 인상이 결정됐습니다. 여기서 UPDATE로 값을 덮어쓰면 과거가 사라집니다 — SCD Type 2는 다르게 움직입니다.

기존 행의 유효기간을 [3/1 ~ 4/1)로 닫고, 새 가격의 행을 [4/1 ~ ∞)로 추가합니다. 값을 지우는 일은 없습니다.

이제 "3월 15일 기준 가격"은 범위 포함 연산자 하나(@>)로 끝납니다. 변경분 재적용도, 스냅샷 탐색도 필요 없이 시점 조회가 일반 조회만큼 쉬워집니다.

PostgreSQL이라면 EXCLUDE USING gist 제약으로 같은 상품의 기간 겹침을 DB 레벨에서 차단할 수 있습니다. "같은 날짜에 유효한 가격이 2개"라는 데이터 오염이 원천적으로 불가능해집니다.

CREATE EXTENSION IF NOT EXISTS btree_gist;  -- 등호(=)를 GiST에서 쓰기 위해

CREATE TABLE product_price (
    product_id   bigint NOT NULL,
    price        numeric NOT NULL,
    valid_during tstzrange NOT NULL,
    -- 같은 상품의 유효기간 겹침을 DB가 차단
    EXCLUDE USING gist (product_id WITH =, valid_during WITH &&)
);

-- 시점 조회: "3월 15일에 유효했던 가격"
SELECT price FROM product_price
WHERE  product_id = 77 AND valid_during @> '2026-03-15'::timestamptz;

-- 현재가 조회
SELECT price FROM product_price
WHERE  product_id = 77 AND valid_during @> now();
⚠️
유효기간 방식의 비용

현재 데이터 조회에 항상 기간 조건이 붙고, 이 테이블을 참조하는 FK 설계가 복잡해집니다(과거 행도 같은 테이블에 있으므로). "시점 조회가 핵심 요구인 데이터"에만 쓰고, 일반 엔터티 전체에 남발하지 마세요.

2-3. 도메인 이벤트 방식 — 사용자에게 보여줄 이력

상태를 저장하는 대신 "발생한 사건"(주문 생성됨, 배송지 변경됨, 환불 요청됨)을 append-only로 쌓는 방식입니다. 감사 로그를 그대로 화면에 뿌리면 내부 컬럼명이 노출되고 문구 가공도 어렵기 때문에, 사용자 노출용 이력은 이렇게 따로 쌓는 경우가 많습니다.

CREATE TABLE order_events (
    event_id   bigserial PRIMARY KEY,
    order_id   bigint NOT NULL,
    event_type text NOT NULL,     -- ORDER_PLACED / ADDRESS_CHANGED / REFUND_REQUESTED …
    actor_id   bigint,
    payload    jsonb,             -- 화면 문구에 필요한 값만: {"from":"서울…","to":"부산…"}
    created_at timestamptz NOT NULL DEFAULT now()
);

-- "홍길동님이 배송지를 서울…에서 부산…으로 변경했습니다" — payload를 템플릿에 바인딩
SELECT event_type, actor_id, payload, created_at
FROM   order_events WHERE order_id = 1042 ORDER BY created_at;
📌
모든 컬럼을 기록할 필요가 없다

도메인 이벤트는 감사 로그와 달리 사용자에게 의미 있는 변경만 골라서 애플리케이션 레벨에서 기록합니다. 완전성은 트리거 기반 감사 로그가 담당하고, 이 테이블은 읽기 편한 이야기를 담당합니다 — 역할 분담입니다.

3. 누가 보는 이력인가 — 용도별 선택 기준

"이력 테이블 어떻게 짜요?"에 대한 정답이 하나가 아닌 이유는, 보는 사람과 목적에 따라 요구 자체가 다르기 때문입니다.

🕵️ 관리자용 / 감사(Audit)

요구: 누락 없이 전부, 조작 불가능하게, 증거 수준으로. 조회는 드묾(사고 났을 때만).

추천: 트리거 + jsonb 하이브리드(2-1). 앱 코드가 아닌 DB 트리거에서 기록해야 직접 SQL 수정 같은 우회 경로까지 잡힙니다. 이력 테이블의 UPDATE/DELETE 권한은 회수.

🙋 사용자 노출용 활동 내역

요구: "배송지를 A에서 B로 변경" — 사람이 읽는 문장. 조회가 잦고 빨라야 함.

추천: 도메인 이벤트(2-3) 또는 앱 레벨 AS-IS/TO-BE. 의미 있는 변경만 선별 기록. 감사 로그와 분리하는 것이 핵심.

📅 시점 조회 핵심 데이터

요구: "3월 15일 기준 이 상품 가격"을 정확히 재현. 요금·환율·계약·조직도.

추천: 유효기간/SCD Type 2(2-2). PostgreSQL이면 range 타입 + EXCLUDE 제약으로 겹침까지 차단.

📊 통계 / 분석

요구: 추세 분석, 대량 집계. 운영 DB 부하 없이.

추천: 일 단위 배치 스냅샷 또는 CDC(Debezium 등)로 웨어하우스·ClickHouse에 적재. 운영 DB에 분석용 이력을 무한 적립하지 않는 것이 핵심.

상황추천 패턴이유
감사/컴플라이언스, "증거"가 필요트리거 + jsonb 스냅샷(old/new)완전성·우회 불가·스키마 변경에 강함
사용자에게 보여줄 변경 내역 UI도메인 이벤트 / 앱 레벨 AS-IS·TO-BE읽기 좋은 문장, 빠른 조회, 선별 기록
특정 시점 값 재현이 핵심유효기간(SCD Type 2) + range 타입시점 조회가 일반 조회만큼 쉬움
요구가 아직 불명확jsonb 하이브리드(2-1)old/new 전체 + changed_cols를 다 보유 — 어느 요구가 와도 대응
💡
실무에서 가장 후회가 적은 구성 — 이원화

"감사 로그는 트리거로 무조건 전부(하이브리드), 사용자 노출용 이력은 필요한 것만 앱에서 따로" — 두 테이블을 분리하는 구성이 가장 흔하고 확장에도 강합니다. 하나의 이력 테이블로 감사·UI·시점조회를 전부 해결하려는 시도가 바로 아래 안티패턴 ①입니다.

4. 안티패턴 6선

❌
① 만능 이력 테이블 — 하나로 모든 용도 해결하기

감사도, 사용자 화면도, 시점 조회도 한 테이블로 처리하면: 감사엔 완전성이 부족하고, 화면엔 내부 컬럼명이 노출되고, 시점 조회는 느립니다. 처방: 용도별 분리(3장). 최소한 감사용과 사용자 노출용은 나누세요.

❌
② 이력 테이블에 원본 FK + ON DELETE CASCADE

원본 행이 삭제되면 이력까지 연쇄 삭제 — "삭제됐다"는 사실을 기록해야 할 이력이 삭제와 함께 증발합니다. 감사 목적이라면 치명적입니다. 처방: 이력 테이블에는 원본 ID를 일반 컬럼으로만 저장하고 FK를 걸지 않습니다. DELETE 자체를 이력 행(operation='D')으로 남기세요.

❌
③ 감사 기록을 애플리케이션 코드에서만 남기기

운영 중 psql로 직접 실행한 UPDATE, 배치 스크립트, 다른 서비스의 접근은 전부 기록에서 빠집니다. 감사 로그의 존재 이유가 무너집니다. 처방: 감사 목적이면 트리거(또는 pgaudit·CDC 등 DB 레벨 수단). 앱 레벨 기록은 사용자 노출용 이력에만.

❌
④ 이력 테이블에 인덱스 잔뜩 걸기

이력 테이블은 쓰기 99% 워크로드입니다. 트리거로 기록한다면 인덱스 유지 비용이 원본 테이블의 모든 트랜잭션을 느리게 만듭니다. 처방: (record_id, changed_at) 정도만. 분석용 조회는 복제본이나 웨어하우스에서.

❌
⑤ 보존 정책 없이 무한 적립

이력 테이블은 원본보다 몇 배 빠르게 자랍니다. 2년 뒤 수억 행에서 "오래된 이력 DELETE"를 시도하면 dead tuple 폭증과 VACUUM 지옥이 기다립니다. 처방: 처음부터 changed_at 기준 RANGE 파티셔닝 — 만료 이력 정리가 DROP PARTITION 한 방으로 끝납니다. (PostgreSQL 12장 파티셔닝 참고)

❌
⑥ delta만 쌓고 시점 복원 요구 받기 (그리고 그 반대)

AS-IS/TO-BE만 쌓았는데 "장애 직전 상태로 복원해 달라"는 요구가 오면, 수만 건의 변경분을 순서대로 재적용해야 하고 diff 추출 로직에 버그가 하나라도 있었다면 복원 결과를 신뢰할 수 없습니다. 반대로 스냅샷만 쌓으면 "누가 뭘 바꿨나" 리포트마다 전 행 비교 연산이 필요합니다. 처방: 어느 질문을 받게 될지 불확실하면 jsonb 하이브리드(2-1)로 둘 다 확보.

5. 설계 체크리스트

  • 용도 정의 — 이 이력은 누가, 어떤 질문에 답하려고 보는가? (감사 / 사용자 화면 / 시점 조회 / 분석)
  • 기록 주체 — 감사면 DB 트리거, 사용자 노출용이면 앱 레벨. set_config로 앱 컨텍스트(사용자 ID)를 트리거에 전달.
  • append-only 보장 — 이력 테이블의 UPDATE/DELETE 권한 회수. PK는 별도 surrogate key.
  • 원본 FK 금지 — 원본 삭제 후에도 이력은 남아야 한다. 원본 ID는 일반 컬럼으로.
  • 인덱스 최소화 — (record_id, changed_at) 수준만. 쓰기 성능이 우선.
  • 트랜잭션 일관성 — 이력 기록은 원본 변경과 같은 트랜잭션에서. (트리거는 자동 보장)
  • 보존 정책 + 파티셔닝 — changed_at RANGE 파티셔닝, 보존 기간 경과분은 DROP PARTITION.
  • 스키마 변경 시나리오 — 원본에 컬럼이 추가되면 이력은 어떻게 되는가? (jsonb 방식은 자동 흡수)

✍️ 이해도 체크

컴플라이언스 요구로 "모든 데이터 변경을 빠짐없이" 기록해야 합니다. 기록 주체로 가장 적절한 것은?
✅ 앱 레벨 기록(서비스 레이어·ORM 훅)은 그 코드 경로를 거치지 않는 변경 — 운영 중 직접 실행한 SQL, 배치 스크립트, 다른 애플리케이션 — 을 전부 놓칩니다. 배치 비교는 하루 안의 중간 변경들이 사라집니다. 완전성이 요구되는 감사는 DB 레벨(트리거, pgaudit, CDC)이 답입니다. 앱 컨텍스트(사용자 ID)는 set_config 패턴으로 전달합니다.
"장애 발생 직전인 어제 14:00 시점의 주문 데이터 전체 상태"를 자주 복원해야 하는 시스템입니다. 유리한 이력 방식은?
✅ 시점 복원은 스냅샷의 홈그라운드입니다 — changed_at <= '14:00' ORDER BY changed_at DESC LIMIT 1 한 번이면 끝. AS-IS/TO-BE는 초기 상태에서 변경분을 순서대로 전부 재적용해야 하고, diff 추출 로직에 버그가 있었다면 복원 결과 자체를 신뢰할 수 없습니다. "정보량이 같다"도 함정: delta 방식은 기록 시점의 추출 로직이 완벽했을 때만 정보가 보존됩니다.
다음 이력 테이블 DDL에서 가장 심각한 문제는? CREATE TABLE orders_history (…, order_id bigint REFERENCES orders(id) ON DELETE CASCADE, …)
✅ ON DELETE CASCADE가 붙은 FK 때문에 원본 행 삭제 시 이력도 연쇄 삭제됩니다. "누가 언제 삭제했나"를 증언해야 할 이력이 삭제와 동시에 사라지는 것 — 감사 목적에 치명적인 안티패턴 ②입니다. 이력 테이블에는 원본 ID를 일반 컬럼으로만 저장하고, DELETE 자체를 이력 행(operation='D')으로 남기세요.
